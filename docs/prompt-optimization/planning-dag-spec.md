# Planning DAG Specification: `cpp-tui-widget-library`

## (1) Purpose and Scope

This DAG is the **first planning optimization scenario** for the ZeptoCode prompt optimization system. It is **NOT run** by the optimization harness; it serves as a **plan quality evaluation target only**.

**Why this DAG:**
- Provides a maximally complex reference DAG against which planning prompt quality can be scored
- Naturally generates the topological complexity needed to stress-test planning workflow prompts: multiple concurrent implementation branches, shared architectural decisions, per-widget rendering tradeoffs, user involvement in concept selection
- Enables the optimization harness to evaluate whether small models (gemma4:4b) can generate high-quality plans when given optimized prompts
- Non-interactive reference allows scoring without execution (no terminal rendering verification needed)

**Role in Optimization:**
- Execution optimization converges first (agent prompts are locked)
- Planning optimization then uses this reference DAG to score candidate planning prompt changes
- The LLM judge compares a small model's generated plan against this reference to evaluate planning prompt quality

## (2) Planning Task Description

### Scenario Identification

- **Scenario name:** `cpp-tui-widget-library`
- **Project:** `optimization-scenarios/cpp-greenfield-project/` (same physical project as execution scenario)
- **Planning task:** Add a **TUI (Terminal User Interface) component library** to tui-demo-project supporting 3–4 distinct widget types with different rendering requirements

### Widget Types and Rendering Complexity

| Widget | Rendering Approach Options | Tradeoff Dimensions |
|--------|---|---|
| **Text Label** | Simple print; styled (ANSI colors); formatted layout | Complexity vs visual polish |
| **Progress Bar** | Simple ASCII bar; colored bar; animated bar | Stateful vs stateless; ANSI codes vs termlib |
| **Table** | ASCII grid; formatted columns; dynamic sizing | Memory efficiency vs visual quality |
| **Interactive Input Box** | Line editing; input validation; special key handling; clipboard support | Feature scope vs implementation effort |

### User Involvement and Complexity

The plan **must involve the user** in two critical decisions:

1. **Widget Selection:** Which widget types to prioritize (all 4, or focus on 2–3)?
2. **Rendering Strategy:** How each widget type should approach rendering — stateful vs stateless, ANSI colors vs a terminal library, shared rendering primitives vs per-widget logic

These are **genuine tradeoffs** with architectural implications for downstream implementation:
- Choosing ANSI codes is simpler but monochrome-unsafe
- Choosing a terminal library (e.g., `termui`, `ncurses` via CPM) adds dependency complexity but enables rich rendering
- Choosing shared rendering primitives reduces code duplication but increases abstraction overhead
- Choosing per-widget rendering simplifies each widget but increases total code volume

**The planning agent should not decide these unilaterally — the user's input is essential.**

## (3) DAG Phase Composition: Topology Requirements

This DAG is designed for **maximum topological complexity**. The exact node-level structure is generated by Sonnet at plan creation time, but the following topology is **required:**

### Required Structural Elements

1. **Entry `collaborate` phase (gates all downstream work):**
   - Involves the user in widget selection and rendering approach
   - Records the user's decisions (which widgets, rendering strategy choice)
   - All implementation branches depend on this decision outcome

2. **Top-level `deliberate` phases with branching (`is-branch=true`):**
   - Architectural decisions that produce **multiple implementation branches**
   - Each branch represents a distinct widget type (text-label, progress-bar, table, input-box)
   - Branches are **independent work permutations** — different implementations, different verification criteria, different documentation scope

3. **Nested `deliberate` phases within each widget branch:**
   - Widget-specific rendering decisions (stateful vs stateless, color support, event handling specifics)
   - Does NOT branch further — each widget has one decision → one implementation path

4. **Sequential sub-DAG per branch:**
   - Order: `deliberate` (widget-specific) → `implement-code` (widget impl) → `verify` (widget testing) → `author-documentation` (widget API docs)
   - Each widget is a complete, self-contained work track

5. **Asymmetric merge topology (critical for topological complexity):**
   - **Widgets that share rendering primitives** (e.g., text-label and progress-bar both use simple ANSI output) **merge at a shared implementation node**
     - This shared node builds common rendering utilities (color support, box drawing, etc.)
     - All merged widgets depend on this shared code
   - **Widgets with independent rendering** (e.g., interactive input-box with special key handling) **remain on separate branches** through to finish
   - This creates a **non-trivial merge DAG**, not a simple fork-join

6. **Multiple user involvement points (if warranted):**
   - Initial `collaborate` phase (widget selection + approach)
   - Potential additional `collaborate` phases if the planning agent determines that user review is needed (e.g., shared interface design review before implementation begins)

### Illustrative DAG Topology

(Note: This is a conceptual illustration. Sonnet generates the actual node IDs and phase structure.)

```
entry-phase
    └── collaborate(phase_0)
        purpose: ["Widget selection", "Rendering strategy (ANSI vs termlib, shared vs per-widget primitives)"]
        is-branch: false
        ↓
    deliberate(phase_1) — "Shared architecture decisions"
        is-branch: true
        ├─→ [branch: text-label]
        │   ├─ deliberate(phase_2a) — "Text rendering: colors, formatting"
        │   ├─ implement-code(phase_3a) — "Implement Label widget"
        │   ├─ verify(phase_4a) — "Test label rendering, color support"
        │   └─ author-documentation(phase_5a) — "Document Label API"
        │
        ├─→ [branch: progress-bar]
        │   ├─ deliberate(phase_2b) — "Progress bar: animation, width, colors"
        │   ├─ implement-code(phase_3b) — "Implement ProgressBar widget"
        │   ├─ verify(phase_4b) — "Test progress animation"
        │   └─ author-documentation(phase_5b) — "Document ProgressBar API"
        │
        ├─→ [branch: table]
        │   ├─ deliberate(phase_2c) — "Table: column sizing, padding, border style"
        │   ├─ implement-code(phase_3c) — "Implement Table widget"
        │   ├─ verify(phase_4c) — "Test table layout and rendering"
        │   └─ author-documentation(phase_5c) — "Document Table API"
        │
        └─→ [branch: input-box]
            ├─ deliberate(phase_2d) — "Input: line editing, validation, clipboard"
            ├─ implement-code(phase_3d) — "Implement InputBox widget"
            ├─ verify(phase_4d) — "Test input handling and events"
            └─ author-documentation(phase_5d) — "Document InputBox API"
                    ↓
                    └─→ [MERGE: shared-interface]
                        ├─ deliberate(phase_6) — "Shared base class / interface design"
                        ├─ implement-code(phase_7) — "Implement shared interface/base class"
                        ├─ verify(phase_8) — "Verify interface compatibility across widgets"
                        └─ author-documentation(phase_9) — "Document shared interface / architecture overview"
                            ↓
                            └─ finish(phase_10)
```

**Topology features:**
- **4 parallel implementation branches** (one per widget)
- **Nested deliberation** within each branch
- **Shared interface merges** some branches (those using common rendering approach)
- **Independent branches** remain separate (input-box with unique event handling)
- **Multiple decision levels** — top-level architecture, then widget-specific, then shared interface

### Alternative Topology (if warranted by planning decisions)

If the planning agent determines that some widgets share more than just rendering primitives (e.g., all non-interactive widgets share a common base), the merge point could be earlier and more branches could converge. The key is that **asymmetry** and **multiple merge points** create the topological complexity.

## (4) Optimization Targets

When this DAG is used as an evaluation target for planning optimization:

### Mutable Surface: Planning Workflow Prompts Only

All 6 files in `files/planning/plan-session/prompts/*.md`:

| Prompt File | Role in Planning Quality | Optimization Impact |
|---|---|---|
| **`session-overview.md`** | Metadata extraction (plan name, user request, user involvement, constraints) | Quality of initial context parsing; downstream decisions depend on accurate metadata |
| **`orientation-scout.md`** | Codebase survey delegation to context-scout agent | Impact on prior-research availability for planning decisions |
| **`external-research.md`** | Conditional delegation of external research (GitHub APIs, terminal libraries, etc.) | Availability of dependency/library info for deliberation decisions |
| **`draft-plan.md`** | **Highest leverage** — DAG TOML creation and `create_plan` invocation | Directly drives DAG structure; determines whether branches, merges, and topological complexity are generated |
| **`store-notes.md`** | Semantic note persistence to Qdrant | Impacts execution phase retrieval quality (planning phase must store context correctly for execution) |
| **`plan-success.md`** | Final plan presentation and user decision | Clarity of generated plan; ability to request modifications |

### Fixed During Planning Optimization

- **All `files/agents/*.md` system prompts** — frozen at their execution-optimization-optimized state
- **All `files/planning/plan-session/node-library/*/prompt.md`** — locked (only node structure, not prompts, can vary based on TOML generation)
- **Node specs, phase schemas** — immutable

### How the Evaluation Works

1. Optimization harness runs `/plan-session` with this planning task using **gemma4:4b** (via naga-ollama profile)
2. Candidate plan (generated `phase-plan.toml`) is captured
3. LLM judge receives:
   - This reference DAG specification (§3 topology requirements)
   - The candidate plan (generated TOML)
4. Judge scores candidate against reference dimensions (see §5)
5. Feedback informs GEPA's next generation of planning prompt mutations

## (5) Plan Quality Metric Definition

### Metric Type: LLM-as-Judge Comparison

- **Judge:** Frontier model (Claude Sonnet or equivalent)
- **Comparison:** Reference DAG specification vs. candidate plan generated by gemma4:4b
- **Scoring dimensions:** Structural validity, topological complexity, semantic completeness

### Scoring Dimensions (Detailed Rubric)

| Dimension | Description | Scoring Criteria | Excellent (0.85–1.0) | Good (0.65–0.85) | Poor (0.35–0.65) | Failure (0–0.35) |
|-----------|---|---|---|---|---|---|
| **Structural Validity** | Is the plan a valid DAG? No cycles, all exits resolve, no orphans? | DAG graph properties | Valid DAG, all nodes reachable, clean exits | Valid DAG, minor dangling references resolved | Some cycle risk or unreachable nodes | Multiple cycles or broken structure |
| **Collaborate Phase Placement** | Does a `collaborate` phase exist early (before implementation begins)? Does it gate downstream work? | Entry point timing; decision sequencing | `collaborate` at entry; all branches depend on it | `collaborate` early but may not strictly gate all branches | `collaborate` present but late (after some deliberation) | No `collaborate` phase or appears at end |
| **Branch Topology** | Are multiple implementation branches present? Do they correspond to logically distinct work (widget types)? | # of branches; semantic alignment to task | 4 branches (one per widget) with clear semantic identity | 3–4 branches but some overlap in purpose | 2 branches or unclear semantic identity | ≤1 branch or branching contradicts task |
| **Nested Deliberation** | Are per-branch `deliberate` phases present for widget-specific decisions? | Deliberation at ≥2 levels (top-level + per-branch) | 2-level deliberation in all 4 branches | 2-level deliberation in 2–3 branches | 1-level only or deliberation in 1 branch | No nested deliberation |
| **Merge Correctness** | Do branches that share rendering primitives converge at a shared implementation node? | Merge point semantics; branch convergence logic | 2–3 branches merge at shared interface node; others remain independent | 2 branches merge; rationale present but could be clearer | Merge attempt present but semantics unclear | No merge; all branches independent to finish |
| **Asymmetric Merges** | Do independent branches (e.g., input-box) remain separate while shared-primitive branches merge? | Topology varies by branch characteristics; not uniform fork-join | Asymmetric merge pattern clearly evident | Mostly asymmetric with 1 exception | Some symmetry but mostly asymmetric | Uniform fork-join (all branches treated identically) |
| **Sequential Phase Discipline** | Does each branch contain an ordered sequence (deliberate → implement → verify → document)? | Temporal ordering within each branch | All 4 branches follow strict sequence | 3 branches follow sequence; 1 out of order | 2 branches follow sequence; phase reordering in others | Phases out of order in multiple branches |
| **Semantic Completeness** | Does the plan cover the full task scope (all selected widgets, shared architecture, documentation)? | Task coverage; no missing work items | All widgets addressed; shared interface designed; docs present | 3–4 widgets addressed; shared interface implied; docs mostly present | 2–3 widgets addressed; shared interface partial; docs incomplete | 1 widget or less; no shared architecture; no docs |
| **Constraint Adherence** | Does the plan respect Pixi/CPM constraints, C++ conventions, and TUI-specific requirements? | Constraint compliance; no policy violations | All constraints (pixi, CPM, C++23, TUI) respected | Minor constraint gaps (e.g., missing research for termlib) | Several constraints overlooked (e.g., CPM not mentioned) | Major constraint violations (e.g., raw system calls in plan) |
| **Collaborate Semantics** | Is user involvement semantics clear? Does the plan specify what decision the user makes? | Collaborate node `purpose` clarity; decision outcome mapping | Purpose lists specific decisions; branches clearly map to user choices | Purpose clear but decision→branch mapping implicit | Purpose vague or decision mapping unclear | No purpose or user choice not reflected in branches |

### Score Calculation

- **score = (weighted average of dimension scores)**
- Recommended weights: Structural Validity (10%), Collaborate Placement (15%), Branch Topology (20%), Nested Deliberation (15%), Merge Correctness (15%), Asymmetric Merges (10%), Sequential Discipline (10%), Semantic Completeness (10%), Constraint Adherence (5%), Collaborate Semantics (5%)
- **Final score:** float [0.0, 1.0]

### Feedback Format

```json
{
  "score": 0.78,
  "feedback": "Strong collaborate phase at entry with clear widget selection and rendering strategy decisions. 4 branches present with appropriate semantic identity (one per widget). Nested deliberation present in 3 of 4 branches (missing in input-box). Merge topology partially asymmetric — text-label and progress-bar converge at shared interface (correct), but table also merges (should remain independent due to different rendering). Overall topological quality is good but merge logic needs refinement. Sequential phases mostly correct. Constraint adherence solid (pixi/CPM mentioned); documentation present in all branches."
}
```

**Key signal in feedback:**
- Specific identification of which topological elements are missing or malformed
- Examples: "no merge node for shared primitives", "deliberate phases missing within widget branches", "collaborate phase appears after implementation start"

## (6) Relationship to Execution Optimization

### Strict Ordering Constraint

**Planning optimization is strictly downstream of execution optimization.**

- Execution optimization must **converge** (agent prompts frozen at their optimized state) before planning optimization begins
- Planning optimization then runs with execution-optimized agents as the stable execution substrate

### Rationale

Planning prompt quality is assessed by running gemma4:4b through the `/plan-session` workflow. The quality of that run depends on correct agent behavior (headwrench orchestration, context-scout research, etc.). If agent prompts are still being mutated during planning optimization, the planning score is **confounded by agent behavior variance** — it becomes impossible to isolate planning prompt quality as the independent variable.

### Handoff Mechanism

1. Execution optimization runs until GEPA frontier converges
   - Node prompts + agent behavioral sections are mutated
   - Frontier is reached (no further improvement)

2. Optimized agent prompts are **committed to git and frozen**
   - `files/agents/*.md` at their final optimized state
   - `files/planning/plan-session/node-library/*/prompt.md` at their final optimized state

3. Planning optimization begins
   - `files/planning/plan-session/prompts/*.md` (the 6 planning workflow prompts) become the **sole mutable surface**
   - Agent prompts remain fixed (input to the planning optimizer)
   - `/plan-session` runs use these optimized agents as the execution substrate

4. Plan quality is scored with stable agent behavior
   - Signal is clean; planning prompt quality is isolated as the independent variable

## (7) Generation Method

The reference `phase-plan.toml` is generated by running `/plan-session` with the **default or copilot profile (Claude Sonnet)** using the following prompt:

```
I want to add a TUI component library to the cpp-demo-project (C++, CMake, Pixi,
C++23). The library should support 3-4 distinct widget types, each with different
rendering requirements: a text label, a progress bar, a table, and an interactive
input box.

I want to be involved in choosing which widgets to prioritize and what the
rendering approach should be for each — there are real tradeoffs here (stateful
vs stateless rendering, ANSI escape codes vs a terminal library, shared rendering
primitives vs per-widget rendering, color support vs monochrome). Different widget
types may warrant different approaches and should be explored independently.

Hard constraints:
- The plan must use a collaborate phase to involve me in widget selection and
  rendering approach decisions before implementation begins.
- Widget types that share rendering primitives should be planned with a merge
  point back to a shared implementation phase; widget types with independent
  rendering should remain on separate branches through to completion.
- Each widget implementation branch should be broken into sequential phases:
  research/deliberate → implement → verify → document.
- Nested deliberation is expected: top-level decisions about shared architecture,
  then per-widget decisions about rendering specifics.
- Some branches should merge (shared interface/base class), others should not
  (widget-specific rendering logic).
- The plan must be as structurally complex as the task warrants — do not simplify
  the DAG topology to make it easier to generate. If the task naturally requires
  nested branches, asymmetric merges, or multiple collaborate phases, include them.
- Pixi for all package management; CPM for any GitHub-only dependencies.
```

### Post-Generation Workflow

1. Run `/plan-session` with the above prompt using the **Sonnet (default/copilot) profile**
2. Copy the generated `phase-plan.toml` to `.opencode/session-plans/cpp-tui-widget-library/phase-plan.toml`
3. **Do NOT capture a Qdrant snapshot** (unlike execution scenarios)
   - Planning scenarios are evaluated fresh on each GEPA generation
   - The planning optimizer runs `/plan-session` each time with the current candidate planning prompts, generating a new plan for scoring
   - No baseline Qdrant state needs to be committed
4. **Commit:** `git add manifest.yaml phase-plan.toml && git commit -m "Add cpp-tui-widget-library planning scenario"`

### manifest.yaml Entry

```yaml
scenarios:
  - name: cpp-tui-widget-library
    type: planning
```

## (8) Key Topological Features for Scoring Reference

When the LLM judge evaluates a candidate plan, these are the specific topological features it should look for:

### Feature 1: Collaborate Gate

```
expected: collaborate(widget_selection, rendering_approach) at entry
         → all downstream branches depend on this

candidate_good: collaborate exists, is-branch=false, appears first
candidate_poor: collaborate appears late or is not gating (parallel to other phases)
```

### Feature 2: Multi-Widget Branching

```
expected: ≥3 branches (ideally 4: label, progress-bar, table, input-box)
         each branch → independent implementation track

candidate_good: branch_0→label, branch_1→progress_bar, branch_2→table, branch_3→input_box
candidate_poor: <3 branches or branches don't map to widgets
```

### Feature 3: Nested Deliberation

```
expected: deliberate(shared_architecture) at top level, is-branch=true
         → [branch_0] deliberate(label_specifics) → implement → verify → document
         → [branch_1] deliberate(progress_bar_specifics) → implement → verify → document

candidate_good: nested deliberation in all 4 branches
candidate_poor: only top-level deliberation or missing per-branch decisions
```

### Feature 4: Asymmetric Merge

```
expected: label & progress_bar → merge @ shared_interface
         table → also merge @ shared_interface (different rendering, but shares base)
         input_box → [INDEPENDENT] no merge (unique event handling)

candidate_good: 2–3 branches merge; 1–2 branches remain independent
candidate_poor: all branches merge identically OR all branches remain independent
```

### Feature 5: Sequential Discipline per Branch

```
expected: each branch = deliberate → implement → verify → document

candidate_good: all 4 branches follow sequence
candidate_poor: phases out of order in some branches
```
